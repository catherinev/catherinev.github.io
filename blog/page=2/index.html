Don't Park There Chicago: the design

See this post for more background on the Don't Park There Chicago

The overall design of Don't Park There Chicago went through 3 distinct phases. The first looked like this:

Design 1

It's pretty snazzy looking. However, it had a few problems. The first was that the background picture was a large file that took a while to load. We could probably have decreased the size of the file, but there was a bigger problem: the design impeded the functionality of the site. At the bottom of the page was a super cool map and more information about street sweeping -- information that is valuable to many users. Users must either scroll down or click the down arrow button to get that information. But user testing revealed that almost no one scrolled or clicked the button. Don't Park There Chicago: the database

See this post for more background on the Don't Park There Chicago

The data that backs Don't Park There Chicago is available in the City of Chicago Data Portal, specifically at this site. The dataset includes all the information we needed to make the site -- geospatial encoding of each street sweeping region and the corresponding street sweeping dates for 2014. Before our app could do anything useful, we had to figure out how to harness the data from the data portal. After doing some research, we decided to store the data into a PostgreSQL database supported by PostGIS, an extension that allows PostgreSQL to store geospatial data and provides query support for that data. Although we hadn't used PostGIS before, the documentation was great, so we were able to figure out how to use it pretty quickly.

However, getting the data from the data portal to our own database did not go smoothly. The data portal provides several different options for the format of the data: Shapefile, KML, and KMZ. Theoretically, it should have been easy to import the Shapefile directly into our database. However, when we imported the Shapefile, none of the geospatial queries yielded the expected results. Part of the issue was that the data in our database was encoded, so it was difficult to tell why the queries weren't behaving as expected.   read more »

Don't Park There Chicago

I officially graduated from DBC just over a week ago. It was so exciting!

For the last 8 days of bootcamp, everyone works on a final project. I was part of a team with 5 awesome people, and we built an app that I think came out pretty great: Don't Park There Chicago. The app informs users of street sweeping dates in their areas and uses geospatial data about Chicago's 888 street sweeping regions to render an interactive Leaflet map. Users may also sign up for email notifications (powered by Mandrill) or text notifications (powered by Twilio) to avoid being ticketed.   read more »

Group Projects

To be perfectly honest, the real reason I want to write this post is so I can show off the amazing website my group made last weekend. I think everyone can agree it has the best web design seen since the late 90s.

As long as I'm writing this post, I might as well take the opportunity to mention group work at DBC. Each week we are assigned a new group of 3-4 people. On Monday, Tuesday, and Wednesday, each group splits into pairs and the pairs work on core challenges. Thursday is assessment day/optional pairing day (after assessment, we can choose to pair or work solo). On Friday, the whole group comes together to work on one big project.
